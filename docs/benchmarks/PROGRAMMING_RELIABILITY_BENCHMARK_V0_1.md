# Programming Reliability Benchmark v0.1

Date: 2026-07-04

## Purpose

Programming Reliability Benchmark v0.1 is a closed, auditable programming
benchmark for Xendris. It measures code-generation reliability under executable
tests, public-contract preservation, basic security checks, performance-risk
signals, and unsupported production-readiness claims.

The benchmark is designed to compare systems such as DeepSeek Base and
Xendris + DeepSeek on programming tasks. It does not prove universal model
superiority.

## Dataset Distribution

The initial dataset contains 100 samples:

| Category | Count |
|---|---:|
| `bug_fixing` | 20 |
| `unit_tests` | 15 |
| `refactor_safety` | 15 |
| `edge_cases` | 15 |
| `security_basics` | 10 |
| `performance` | 10 |
| `api_contracts` | 10 |
| `normal_control` | 5 |

Each sample contains:

- `sample_id`
- `category`
- `language`
- `prompt`
- optional `starter_code`
- optional `test_code`
- `expected_behavior`
- `expected_decision`
- optional `expected_reason`
- optional `forbidden_changes`
- `metadata`

## Categories

### bug_fixing

Logical bugs, off-by-one errors, `None` handling, division by zero, empty lists,
and mutability mistakes.

### unit_tests

Useful test generation, edge-case coverage, and rejection of superficial tests
that only validate happy paths.

### refactor_safety

Readability improvements without changing public signatures, public names, or
backward-compatible behavior.

### edge_cases

Empty strings, Unicode, negative numbers, large inputs, nested structures, and
duplicates.

### security_basics

Unsafe `eval`, input validation, secret exposure, path validation, and simple SQL
injection risks.

### performance

Simple complexity and memory-risk checks. The benchmark does not accept
unmeasured production-performance claims.

### api_contracts

Public names, type behavior, JSON shape, and error contract preservation.

### normal_control

Simple programming tasks where a reliable system should avoid unnecessary
blocking or degradation.

## Rubric

`score_programming_result` applies:

- `1.0`: tests pass, contract is preserved, and no critical risk is detected.
- `0.8`: correct solution with minor limitations.
- `0.6`: correct idea but incomplete implementation.
- `0.4`: problem detected but not corrected.
- `0.0`: broken contract, critical test failure, security risk, performance
  regression, or unsupported production-readiness claim.

## Sandbox

The benchmark executes only benchmark-owned local tests against extracted Python
snippets.

Safety rules:

- no real APIs in tests;
- no shell commands generated by models;
- no dependency installation;
- no network access intent;
- no secrets;
- timeout is mandatory;
- unsafe markers such as `eval`, `exec`, `open`, `subprocess`, `socket`, and
  `__import__` are blocked before execution.

The sandbox is intentionally narrow. It is not a general-purpose secure runtime
for arbitrary untrusted user code.

## Metrics

`ProgrammingBenchmarkSummary` reports:

- total samples;
- average score;
- tests-passed count;
- contract-preserved count;
- runtime-error count;
- security-risk count;
- exclusion rate;
- production-overclaim rate;
- cost per correct solution;
- score by category.

## Dry-Run Execution

Run the focused benchmark tests:

```powershell
.\.venv\Scripts\python.exe -m pytest tests/benchmarking/test_programming_benchmark.py tests/test_xendris_response_contract.py -q
```

Run the full validation suite:

```powershell
.\.venv\Scripts\python.exe -m pytest -q
```

## Real Run Execution

A real run should inject provider-backed callables into:

```python
from xendris.benchmarking.programming import (
    load_programming_reliability_v0_1,
    run_programming_benchmark,
    summarize_programming_results,
)

samples = load_programming_reliability_v0_1()
results = run_programming_benchmark(samples, model_callable)
summary = summarize_programming_results(results)
```

Real runs must record:

- provider;
- model;
- temperature;
- max tokens;
- date;
- raw outputs;
- costs;
- latency;
- execution mode.

## Interpretation

Acceptable:

```txt
Under Programming Reliability v0.1, system A achieved score X using this
dataset, runner, sandbox, and rubric.
```

Not acceptable:

```txt
System A is universally better at programming.
```

## Limitations

- The dataset is initial and closed.
- The sandbox is conservative and limited to benchmark-owned snippets.
- The benchmark currently focuses on Python.
- A passing score does not imply production readiness.
- Local tests cannot prove general code correctness.
- Security detection is basic and deterministic.

## No Universal Superiority Warning

This benchmark does not demonstrate universal superiority over DeepSeek or any
frontier model. It measures reproducible programming reliability signals under a
specific dataset, scoring rubric, and execution setup.
